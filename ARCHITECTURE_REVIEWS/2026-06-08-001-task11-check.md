## 三、業務代碼行業標準評估

### 3.1 評估方法

基於已讀源碼，從**專業架構師**視角評估，**不誇張、不挑刺**。  
對標框架：Spring Boot（Java）、Laravel（PHP）、Django（Python）。

### 3.2 ✅ 優秀的設計（業界標準）

#### 3.2.1 Service 層「委託模式」實現標準

```go
type ContentController struct {
    common.BaseController
    svc svc.ContentService  // 接口注入
}
```

**優點**：
- ✅ 符合 Go 介面慣用
- ✅ 易於 mock 進行單元測試
- ✅ Controller 和 Service 職責清晰分離

**業界對標**：與 Spring Boot `@Service` + `@Autowired` 同等水準

---

#### 3.2.2 單字段更新白名單（安全實踐）

```go
var allowedSingleFields = map[string]bool{
    "status": true, "istop": true, "isrecommend": true,
    "isheadline": true, "sorting": true, "title": true,
}

func (s *ContentService) UpdateSingleField(id int, field, value string) error {
    if !allowedSingleFields[field] {
        return errors.New("field not allowed: " + field)
    }
    return model.DB.Model(&model.Content{}).Where("id = ?", id).Update(field, value).Error
}
```

**優點**：
- ✅ 防止 SQL 注入繞過
- ✅ 防止越權字段修改（如 `password`、`id`）
- ✅ 業界標準做法

**業界對標**：與 Laravel `$fillable`、Rails `attr_accessible` 同等

---

#### 3.2.3 模板引擎 Context 抽象

```go
type Context struct {
    Sort        *model.ContentSort
    Content     *model.Content
    Site        *model.Site
    Company     *model.Company
    Page        map[string]interface{}
    Member      *model.Member
    Keyword     string
    CurrentPage int
}
```

**優點**：
- ✅ 統一的渲染上下文
- ✅ 解耦 Controller 和模板
- ✅ 易於測試

**業界對標**：與 React Context、Vue Provide/Inject 概念一致

---

#### 3.2.4 Provider 模式（標籤註冊器）

```go
p.Register("list", func(tagName string, params map[string]string, inner string) string { ... })
```

**優點**：
- ✅ 易於擴展新標籤
- ✅ 解耦標籤實現和核心引擎

**業界對標**：與 Spring 的 BeanDefinitionRegistry、Vue 的插件機制同思路

---

### 3.3 ⚠️ 需要改進的地方（不誇張）

#### 3.3.1 Service 層「無接口化」

**當前寫法**：
```go
type ContentService struct{}  // 結構體，不是介面
func (s *ContentService) ListContents(...) {...}
```

**業界標準應該是**：
```go
type ContentService interface {
    ListContents(...) ([]model.Content, int64, error)
    GetContent(...) (*model.Content, error)
    // ...
}

type contentService struct{}  // 小寫，私有實現
func NewContentService() ContentService {
    return &contentService{}
}
```

**影響**：
- ❌ 無法 mock 進行 controller 單元測試
- ❌ 無法在不修改 controller 的前提下替換實現

**建議改進**：定義 `ContentService` 介面 + `NewContentService()` 構造函數。

**優先級**：⭐⭐⭐⭐ 高

---

#### 3.3.2 Provider 內直接訪問 `model.DB`（不夠純粹）

**當前寫法**：
```go
p.Register("list", func(...) string {
    query := model.DB.Where("status = 1")  // ← Provider 內直接調 DB
    // ...
})
```

**業界標準應該是**：
- Provider 只負責「渲染邏輯」
- 數據查詢應在 Service 層完成
- Provider 接收結構化數據，不直接查 DB

**影響**：
- ❌ Provider 無法離線測試
- ❌ 業務邏輯和模板邏輯混雜

**建議改進**：把 `model.DB` 查詢抽到 Service，Provider 只渲染傳入的數據。

**優先級**：⭐⭐⭐ 中

---

#### 3.3.3 模板特殊字符處理不一致（部分 XSS 防護）

**當前寫法**：
```go
p.Register("keyword", func(...) string {
    return html.EscapeString(ctx.Keyword)  // ← 對 keyword 做了 XSS 防護
})
// 但其他字段（如 content、title）直接返回 ctx.Content.Title
```

**業界標準應該是**：
- 所有用戶輸入字段都應做 XSS 防護
- 或明確區分「safe 字段」和「unsafe 字段」

**影響**：
- ⚠️ 部分 XSS 防護，但不完整

**建議改進**：在 `getContentField` 中對用戶輸入字段統一做 `html.EscapeString`。

**優先級**：⭐⭐⭐⭐ 高（安全問題）

---

#### 3.3.4 錯誤處理不夠細粒度

**當前寫法**：
```go
sorts, _ := csc.svc.ListSorts()  // ← 忽略了錯誤
```

**業界標準應該是**：
```go
sorts, err := csc.svc.ListSorts()
if err != nil {
    cc.JSONFail(c, "Failed to load sorts: " + err.Error())
    return
}
```

**影響**：
- ❌ 數據庫錯誤被靜默吞掉
- ❌ 用戶看到「空白頁」但沒有錯誤提示
- ❌ 調試困難

**建議改進**：所有 Service 調用都應檢查錯誤。

**優先級**：⭐⭐⭐ 中

---

#### 3.3.5 Context 全域可變狀態

**當前寫法**：
```go
type Context struct {
    Page map[string]interface{}  // ← 在 Provider 中修改 ctx.Page
}
```

**影響**：
- ⚠️ 在 `list` provider 中寫入 `ctx.Page["current"]`、`ctx.Page["count"]` 等
- ⚠️ 同一個 Context 在多個 Provider 間共享，可能有並發問題
- ⚠️ 隱式依賴（讀者要追蹤誰寫了 Context）

**建議改進**：
- Provider 應返回純值，不修改 Context
- 或明確標註「哪些 Provider 是 write」

**優先級**：⭐⭐ 低（暫時不影響功能）
